# 任务

分析三个 DDL 操作对存储层的影响:① 建表 ② 表增加字段 ③ 表增加索引。

**「上层 DDL 动作 → 存储层具体发生了什么变化」**这条因果链,落到源码方法级。

# 分析框架:每个操作必须回答的「存储层影响清单」

逐项核对以下 6 个维度,无影响也要明确说明"无";避免挑软柿子捏:

1. **数据结构层**:动了哪些 B+Tree / 数据页 / FreeList?是否新建结构?
2. **数据迁移**:是否回填 / 重写存量数据?范围多大?同步还是后台任务?
3. **Schema / 元数据**:schemaId / 列偏移表 / QueryDescriptor 怎么变?
   旧数据如何兼容读取?
4. **WAL**:记录了哪些 record 类型?Checkpoint 与之如何配合?
5. **分布式协调**:走 `CacheConfiguration` 传播,还是 `SchemaProposeDiscoveryMessage`?
   是否触发 PME?各节点如何达成一致?
6. **并发与失败**:DDL 进行中正在进行的读写怎么办?中途失败如何回滚?

# 三个操作要特别挖深的点

- **加字段**:Ignite 是 **schema-on-read**(不重写存量数据页)。
  重点讲「换 schemaId + 列偏移表 + 旧页兼容读」在 `CacheDataRowStore` 层怎么落地;
  讲清"为什么不需要搬数据"。
- **加索引**:这是存储层动作量最大的操作。
  重点讲「新建一棵 `CacheDataTree` + 对存量数据 **rebuild 回填** +
  回填期间与 WAL / Checkpoint / 在线读写的交互」。
- **分布式路径差异(贯穿三者的核心知识点)**:突出
  建表(随 `CacheConfiguration` 传播) **vs** ALTER/INDEX
  (走 `SchemaProposeDiscoveryMessage`)是**两条不同的分布式路径**(参见 `02:379`),
  并说明各自为何如此。
