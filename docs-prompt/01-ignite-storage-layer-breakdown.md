# 任务

`docs-research/03-ignite-storage-layer.md` 是对 Ignite 2.17.0 存储层的源码级深挖,
内容准确但认知密度极高。本任务不是"把它改简单",而是为它搭建一条阶梯:
产出一组由浅入深的引导文档,让读者读完之后,能回头把 03 文档真正看懂。
定位:新内容是"梯子",03 文档是"天花板"。

# 读者画像(决定详略与类比选材)

- 普通 Java 工程师:常见语法、集合、泛型这些不用解释。
- 但存储引擎是空白:不知道"页/换出/WAL/Checkpoint/亲和性"这些数据库内部机制,
  这是本系列要补的"心智模型"。
- 遇到非显然的 Java 底层机制(堆外 DirectBuffer / NIO、CAS 与 Atomic*、
  ReentrantReadWriteLock、对象内存布局等),该解释就顺手解释一句,
  不要默认读者已经懂。

# 宏观结构:阶梯式(问题驱动,每阶长自上一阶)

开篇先用 03 文档 §0 的"一句话心智模型" + §2 全景图当"地图",让读者先有个骨架。
然后按"要解决的问题"的依赖顺序,拆成 6 阶,每阶一篇短文(放 `docs-research/storage-layer/`):

1. 为什么要"页"和"堆外"——内存怎么管            (对应 03 §4)
2. 一页之内怎么放下变长的行——数据页/link/FreeList  (对应 03 §5.3–5.5)
3. 怎么按键快速找到那行——B+Tree,叶子只存 link     (对应 03 §5.1–5.2)
4. 崩溃了怎么不丢数据——WAL + Checkpoint          (对应 03 §6)
5. 单机装不下、还要高可用——分区+亲和性+rebalance   (对应 03 §7)
6. 串起来:一次 put/get 全链路                    (对应 03 §9)

要求:每一阶开头先用一两句点出"上一阶留下的、本阶要解决的矛盾",
让概念像长出来的一样咬合。(并发/三道锁等横切话题,融进相关阶段讲,不单列。)
篇幅不设固定字数,以"讲透"为准;但每阶只聚焦一个子系统,不要溢出到别的阶段。

# 微观结构:每个知识点统一用"五段式"

1. **痛点**:如果没这个东西,会出什么问题?(先制造需求)
2. **类比**:用读者熟悉的东西类比(TreeMap、git commit、append-only 日志、
   操作系统虚拟内存/页缓存、ByteBuffer 池……)。
3. **原理**:它怎么工作——这一段不要源码行号,优先用图(见下)。
4. **为什么这么设计**:对比一种替代方案,说明取舍。
5. **代码锚点**:最后才给 03 文档里的 `file:line`,作为"原理落地"的印证,点到为止,不逐行展开。

# 图表优先(对读者友好,很重要)

- 凡是能用图说清的原理,就不要只用文字。优先级:**图/表 > 文字**。
- 目标:读者哪怕跳读文字,光看图也能抓住大意。
- 按内容选用形式:
  - **mermaid 流程图/时序图/状态图**——讲流程、讲多模块协作;
  - **ASCII 结构图**——讲内存/字节级布局(pageId 位编码、数据页槽位表、行字节顺序等);
  - **对比表**——讲"为什么这么设计"时对比替代方案,或做术语速查;
  - **类比映射表**——把 Ignite 概念一一映射到读者熟悉的东西。
- 每个知识点的"原理"段尽量配一张图。

# 约束

- 先原理、后代码;代码永远是印证,不是主角。
- 术语第一次出现必须用一句大白话定义,之后才允许直接用。
- 同一时间只引入一个新概念。
- 每篇结尾给两个小栏目:
  - **"你现在应该能回答"**:3 个问题,供自检。
  - **"对应到 03 文档"**:指明覆盖了 03 的哪几节,鼓励读者去读原文印证。

# 验收标准

- 一个普通 Java 工程师,读完每一阶,能用自己的话复述
  "这阶解决什么问题、怎么解决、为什么这样"。
- 不堆术语、不逐行贴源码;图表充足,每阶至少配 2–3 张图/表。
